# Scrum 工作流（建议版）

迭代周期：每个 Sprint 2–4 周；持续交付可验收的增量（Increment）。

## 前置（可选，Sprint 0 或持续发现）
- 业务/市场洞察（持续进行）
- 产出/更新 PRD 摘要与关键验收标准（拆分为 Product Backlog Items）
- 技术可行性与架构 Spike（作为 Enabler/Spike 条目进入待办；必要时在 Sprint 0 时间盒完成）

## 常规迭代（每个 Sprint）
1) Product Backlog 精炼（持续进行）
- 明确优先级、拆分为可交付的故事/任务，补充验收标准与估算

2) Sprint Planning（时间盒）
- 定义 Sprint Goal
- 选择并承诺 PBIs（结合容量与速度）
- 输出 Sprint Backlog 与初步计划

3) 执行与协作
- 开发/测试/集成，持续交付满足 DoD 的增量（Increment）
- Daily Scrum（每日 ≤15 分钟）同步进度与阻碍

4) Sprint Review
- 向干系人展示增量，收集反馈，调整 Product Backlog

5) Sprint Retrospective
- 复盘过程与实践，形成可执行改进项，进入下个 Sprint

## 关键工件与规范
- 工件：Product Backlog、Sprint Backlog、Increment
- 就绪/完成定义：DoR/DoD（含代码规范、测试覆盖、i18n/a11y、隐私合规）
- 估算与度量：Story Points、速度（Velocity）、燃尽/燃起图

## 备注
- 可行性分析与 HLD 非 Scrum 必选事件；推荐以 Spike/Enabler 的形式纳入 Sprint，或在 Sprint 0 时间盒产出轻量版本，并在后续迭代持续演进。
