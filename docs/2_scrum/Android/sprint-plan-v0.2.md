# Prism Player Android Sprint 计划 v0.2

> 版本：v0.2  
> 日期：2025-10-25  
> 关联文档：`docs/requirements/prd_v0.2.md`、`docs/tdd/Android/hld-android-v0.2.md`、`docs/scrum/scrum.md`

## 0. 文档说明

本文档基于 Scrum 敏捷开发框架，为 Prism Player Android 端规划迭代路线图。每个 Sprint 周期为 2–4 周，产出可验收的增量（Increment）。

### 关键原则
- **用户故事优先**：每个任务关联 PRD 中的用户故事与验收标准（AC）
- **技术债务可控**：架构决策（ADR）与重构纳入 Sprint，不积累技术债
- **持续交付**：每个 Sprint 必须产出可演示的增量
- **度量驱动**：关键指标（首帧时间、RTF、同步偏差）在每个 Sprint 中验证

---

## 1. 项目概览

### 1.1 目标与范围
按照 PRD v0.2 与 HLD v0.2，实现一款本地音视频播放器，核心能力：
- 离线 ASR（语音转文字）+ 实时字幕渲染
- 可选离线翻译（首期 en→zh-Hans）
- 字幕导出（SRT/VTT）与样式管理
- 模型管理（下载/导入/校验/删除）

### 1.2 技术栈（摘自 HLD）
- **语言**：Kotlin
- **UI**：Jetpack Compose
- **架构**：MVVM + UseCase 分层
- **核心库**：Media3(ExoPlayer)、Room、Hilt、Coroutines/Flow、WorkManager
- **NDK**：Whisper.cpp/Vosk（ASR）、ONNX Runtime Mobile（翻译）

### 1.3 成功指标（KPI）
| 档位 | 首帧字幕（P95） | RTF | 同步偏差 |
|------|----------------|-----|---------|
| 高端 | ≤5s | ≥1.0 | ≤±200ms |
| 中端 | ≤8s | ≥0.5 | ≤±200ms |
| 入门 | ≤12s | ≥0.3 | ≤±200ms |

## 2. Sprint 总览（里程碑映射）

| Sprint | 周期 | 里程碑 | 核心目标 | 交付物 |
|--------|------|--------|---------|--------|
| **Sprint 0** | 1周 | 基础设施 | 项目脚手架、CI/CD、开发规范 | 可编译运行的骨架应用 |
| **Sprint 1** | 3周 | M1 原型（前半） | 播放器集成、音频提取、基础 UI | 能播放视频的 App |
| **Sprint 2** | 3周 | M1 原型（后半） | NDK ASR、首帧字幕、SRT 导出 | 能显示实时字幕的原型 |
| **Sprint 3** | 3周 | M2 可用版（前半） | 模型管理、滚动识别、缓存策略 | 完整的识别流程 |
| **Sprint 4** | 3周 | M2 可用版（后半） | 翻译原型、倍速适配、VTT 导出 | 支持翻译的可用版 |
| **Sprint 5** | 4周 | M3 优化版 | 性能优化、质量感知、稳健化 | 生产就绪版本 |

## 9. Definition of Ready (DoR)

每个 PBI 进入 Sprint 前必须满足：
- [ ] 用户故事清晰，验收标准明确
- [ ] 估算完成（Story Points）
- [ ] 技术依赖已识别
- [ ] 设计文档可用（如 UI mockup、架构图）
- [ ] 无阻塞性依赖

---

## 10. Definition of Done (DoD)

每个 PBI 完成时必须满足：
- [ ] 代码实现符合验收标准（AC）
- [ ] 单元测试覆盖率 ≥70%（关键模块 ≥80%）
- [ ] UI 测试覆盖核心路径
- [ ] 代码通过 CI（构建、测试、Detekt、ktlint）
- [ ] Code Review 完成（至少 1 人 Approve）
- [ ] 文档更新（如 API 文档、README）
- [ ] 无未解决的 P0/P1 Bug
- [ ] 国际化字符串完整（中英文）
- [ ] 符合可访问性基线（TalkBack 可用）

## 11. 估算与速度（Velocity）

### 11.1 Story Points 定义
- **1 SP**：简单任务，1–2 天完成（如配置文件、简单 UI）
- **2 SP**：中等任务，2–3 天（如一个 UseCase 实现）
- **3 SP**：复杂任务，3–5 天（如数据库设计）
- **5 SP**：较大任务，5–8 天（如 NDK 集成）
- **8 SP**：大型任务，需拆分为子任务（如完整功能模块）

### 11.2 团队容量（假设 2 人团队，3 周 Sprint）
- **每人每周产能**：~8 SP
- **3 周 Sprint 总容量**：2 人 × 3 周 × 8 SP = **48 SP**
- **安全系数**：0.8（考虑会议、代码审查、技术债）
- **实际承诺容量**：~**38 SP**

### 11.3 调整建议
- 前 2 个 Sprint 保守估算，建立 Baseline
- 每个 Sprint Retrospective 后调整速度

## 13. 度量与监控

### 13.1 Sprint 级指标
- **Velocity**：每个 Sprint 完成的 Story Points
- **Burndown Chart**：剩余工作量趋势
- **Defect Density**：发现 Bug 数 / Story Points

### 13.2 产品级指标
- **首帧时间（P95）**：每个 Sprint 在目标设备上验证
- **RTF**：识别速度 / 播放速度
- **同步偏差**：字幕与播放位置偏差
- **崩溃率**：每千次启动崩溃数

### 13.3 工具
- **Jira/Linear**：任务跟踪
- **GitHub Projects**：看板视图
- **Firebase Crashlytics**：崩溃监控（可选）

## 15. 附录

### 15.1 术语表
- **RTF**（Realtime Factor）：识别速度与播放速度的比值，>1 表示快于实时
- **Window**：时间窗口（10–30s），按优先级调度识别
- **Preemptive**：抢占式，优先级最高的任务
- **LRU**：Least Recently Used，最近最少使用缓存淘汰策略
